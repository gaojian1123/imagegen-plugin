# Repository conventions

- **TypeScript source vs bundle:** the server lives in `mcp-server/*.ts` — tool/
  resource wiring in `server.ts` (`buildServer`), image + IO helpers in `client.ts`,
  the stdio entry in `index.ts`. The committed `mcp-server/dist/index.js` is the
  esbuild bundle generated by `npm run build` — never edit it by hand. The App UI
  lives in `mcp-server/ui/` (`mcp-app.html` + `app.ts`) and is bundled into the
  committed `mcp-server/dist/mcp-app.html` by Vite (`vite.config.ts`, part of
  `npm run build`) — also never hand-edit. Rebuild and commit both dist artifacts
  whenever `mcp-server/` changes, and run `npm run typecheck` (`tsc --noEmit`)
  before committing. Dev-only harness (fake images, no Azure): `npm run dev:check`
  (headless) and `npm run dev:host` (Streamable HTTP for a browser host).
- **Tests:** `npm test` runs `node --test mcp-server/client.test.ts` using Node 24's
  native TypeScript type stripping (no `tsx`/`ts-node`). No network calls in tests;
  inject a fake client (cast `as unknown as OpenAI`) for the SDK wrappers. Add a test
  for any non-trivial logic.
- **Dependencies:** runtime deps are `openai`, `@modelcontextprotocol/sdk`,
  `zod`, `@azure/identity` (keyless Entra ID auth), and
  `@modelcontextprotocol/ext-apps` (MCP Apps UI helpers, used server- and
  UI-side). Keep it minimal; `esbuild`, `typescript`, `@types/node`, `vite`, and
  `vite-plugin-singlefile` are dev-only. Vite bundles the UI into a single
  self-contained HTML; esbuild bundles the server.
- **Secrets:** never commit credentials or put them in `.mcp.json`. The server reads
  `AZURE_OPENAI_*` from the environment.
- **Env var names:** `AZURE_OPENAI_ENDPOINT`, `AZURE_OPENAI_API_KEY`,
  `AZURE_OPENAI_IMAGE_DEPLOYMENT`. Output directory is a per-call `output_dir`
  tool argument, not an env var; omit it to keep the image in-app only (no file
  written), pass it to save to disk.
